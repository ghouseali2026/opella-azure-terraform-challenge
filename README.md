## Design Decisions

### Resource Groups vs Subscriptions for Multiple Environments

| Approach | When to use |
|---|---|
| **Resource Groups** (used here) | Same subscription, smaller team, simpler billing, lower operational overhead. Adequate for most projects. |
| **Subscriptions** | Strict blast-radius isolation, independent RBAC boundaries, large enterprise, hard billing separation between BUs. |

For most companies a Resource Group per environment (enforced via naming + tags) provides sufficient isolation without the operational overhead of managing multiple Azure subscriptions.

### Naming Convention

All resources follow: `{project}-{environment}-{region_short}-{resource_type}`

Example: `myapp-dev-eus-vnet`

This makes the environment and region immediately apparent in the Azure portal, in logs, and in alerts — without needing to open resource tags.

### Tagging Strategy

Every resource inherits these tags (enforced via `locals.tags`):

| Tag | Purpose |
|---|---|
| `environment` | Filter by env in Azure Cost Management |
| `project` | Attribute cost/usage to a project |
| `region` | Explicit region label |
| `managed_by = terraform` | Indicates IaC ownership; ops must not manually edit |
| `cost_center` | Finance chargeback |

To **enforce** tags organisation-wide: use **Azure Policy** with a `deny` effect that requires mandatory tags on resource creation. This can be defined in Terraform using `azurerm_policy_definition` and `azurerm_policy_assignment`.

## Module: `modules/vnet`

### What it creates

- Azure Virtual Network
- Configurable subnets (any number)
- Optional Network Security Groups (one per subnet, auto-associated)
- Optional DDoS Protection Plan attachment
- Optional service endpoints and subnet delegations

### Key inputs

| Variable | Required | Description |
|---|---|---|
| `name` | ✅ | VNET name (validated with regex) |
| `resource_group_name` | ✅ | Target resource group |
| `location` | ✅ | Azure region |
| `address_space` | ✅ | List of CIDR ranges |
| `subnets` | ❌ | Map of subnet definitions (see below) |
| `create_nsgs` | ❌ | Create & associate NSGs (default: `true`) |
| `dns_servers` | ❌ | Custom DNS IPs (default: Azure DNS) |
| `ddos_protection_plan_id` | ❌ | Attach existing DDoS plan |
| `tags` | ❌ | Resource tags |

#### Subnet definition shape

```hcl
subnets = {
  my-subnet = {
    address_prefixes  = ["10.0.1.0/24"]          # required
    service_endpoints = ["Microsoft.Storage"]     # optional
    delegation = {                                # optional
      name                       = "app-service"
      service_delegation_name    = "Microsoft.Web/serverFarms"
      service_delegation_actions = ["Microsoft.Network/virtualNetworks/subnets/action"]
    }
    nsg_rules = {                                 # optional, used when create_nsgs=true
      allow-ssh = {
        priority                   = 100
        direction                  = "Inbound"
        access                     = "Allow"
        protocol                   = "Tcp"
        source_port_range          = "*"
        destination_port_range     = "22"
        source_address_prefix      = "10.0.0.0/8"
        destination_address_prefix = "*"
      }
    }
  }
}
```

### Outputs

| Output | Description |
|---|---|
| `vnet_id` | Resource ID — use for VNET peering |
| `vnet_name` | VNET name |
| `vnet_address_space` | Address space list |
| `subnet_ids` | `map(name → id)` — pass to VMs, PaaS, Private Endpoints |
| `subnet_address_prefixes` | `map(name → prefixes)` |
| `nsg_ids` | `map(name → nsg_id)` |

### Documentation generation

Module docs are auto-generated by [`terraform-docs`](https://terraform-docs.io/) via the `pre-commit-terraform` hook. On every commit, `terraform-docs` reads `variables.tf` and `outputs.tf` and rewrites the `README.md` in the module directory.

```bash
# One-time manual generation
terraform-docs markdown table --output-file README.md modules/vnet/
```

## Environments

### Resources deployed (dev)

| Resource | Purpose |
|---|---|
| Resource Group | Isolation boundary |
| VNET + Subnet | Private networking |
| NSG with SSH/RDP rules | Restrict inbound access to admin IP only |
| Storage Account (LRS) + Blob container | Artifact/config storage, locked to VNET via service endpoint |
| Linux VM (`Standard_B1s`) | Developer jump box / build agent |
| Managed Identity on VM | Passwordless access to Storage (RBAC: Storage Blob Data Reader) |

### Resources deployed (prod)

Same foundation but with GRS storage replication, wider subnets (app + data), tighter NSG defaults, and no public VM.

## CI/CD Pipeline

### Release lifecycle

```
feature-branch
     │
     ▼
  Pull Request ──► lint (fmt, tflint, tfsec)
                ──► plan dev   (posted as PR comment)
                ──► plan prod  (posted as PR comment)
     │
     ▼
  Merge to main
     │
     ▼
  apply dev ──► auto-applies (no manual gate)
     │
     ▼
  apply prod ──► waits for manual approval in GitHub Environments
```

**GitHub Environments** (`dev`, `prod`) are configured in repo Settings → Environments:
- `dev`: No required reviewers — fast feedback loop.
- `prod`: Requires 1 reviewer approval before apply runs.

### Authentication to Azure (OIDC — no stored secrets)

The pipeline uses [Azure Workload Identity Federation](https://learn.microsoft.com/en-us/azure/active-directory/workload-identities/workload-identity-federation) (OIDC). No `ARM_CLIENT_SECRET` is ever stored. Only three GitHub secrets are needed:

```
AZURE_CLIENT_ID
AZURE_TENANT_ID
AZURE_SUBSCRIPTION_ID
```

## Module Tests (Terratest)

Tests are written in Go using [Terratest](https://terratest.gruntwork.io/). They provision real Azure resources, assert outputs and Azure API state, then destroy.

```bash
cd tests
go mod download
# Requires an authenticated Azure session (az login or env vars)
go test -v -timeout 30m ./...
```

Two tests:
- `TestVnetModuleDeployment` — standard deploy with NSGs, checks outputs and Azure API.
- `TestVnetModuleNoNSGs` — verifies `create_nsgs=false` produces empty `nsg_ids` output.

## Local Usage

```bash
# Authenticate
az login
az account set --subscription <subscription-id>

# Deploy dev
cd environments/dev
terraform init
terraform plan -var="vm_admin_ssh_public_key=$(cat ~/.ssh/id_rsa.pub)"
terraform apply -var="vm_admin_ssh_public_key=$(cat ~/.ssh/id_rsa.pub)"
```

> **Never commit secrets.** Pass `vm_admin_ssh_public_key` via `-var`, a `.tfvars` file excluded from git (`.gitignore`), or the `TF_VAR_vm_admin_ssh_public_key` environment variable.
